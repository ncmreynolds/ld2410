# 06 — LD2410C extensions (also exposed on LD2410B)

Methods exposed on the LD2410C variant. Every method on this page is
also available on the LD2410B (the B is a wire-level superset of the
C — same opcodes, same envelopes). To make these visible at compile
time, include `<ld2410c.h>` (or `<ld2410b.h>`) instead of
`<ld2410.h>` — equivalently, define `LD2410_VARIANT_C` /
`LD2410_VARIANT_B` before `#include <ld2410.h>` or pass it as a
build flag (`-DLD2410_VARIANT_C`). See
[`02-variants.md`](02-variants.md#selecting-the-variant-at-build-time)
for the comparison between the two routes.

For methods on **all** variants see [`04-api-core.md`](04-api-core.md).
For methods on base + B + C see [`05-api-ld2410-base.md`](05-api-ld2410-base.md).
For the two **B-only** methods that don't apply to the C
(`setAuxiliaryControl` + `requestAuxiliaryControl`, controlling the
B's on-board photodiode) see
[`02-variants.md#status-of-ld2410b-support`](02-variants.md#status-of-ld2410b-support).

The C extensions cluster around features the original LD2410 lacked:

- **UART baud rate** (HLK §2.2.9, opcode 0xA1)
- **Bluetooth** on/off, MAC, password, permissions (HLK §2.2.12-15, opcodes 0xA4 0xA5 0xA8 0xA9)
- **Distance resolution** 0.75 m ↔ 0.20 m per gate (HLK §2.2.16-17, opcodes 0xAA 0xAB)

## Baud rate

### `bool setBaudRate(uint16_t baud_index)`

Change the radar's UART baud rate. **Non-volatile and only takes
effect after `requestRestart()`.** After the restart you must reopen
your local `RADAR_SERIAL.begin(...)` at the new baud.

Use the predefined constants:

| Constant | Baud |
|---|---|
| `LD2410_BAUD_INDEX_9600`    | 9600   |
| `LD2410_BAUD_INDEX_19200`   | 19200  |
| `LD2410_BAUD_INDEX_38400`   | 38400  |
| `LD2410_BAUD_INDEX_57600`   | 57600  |
| `LD2410_BAUD_INDEX_115200`  | 115200 |
| `LD2410_BAUD_INDEX_230400`  | 230400 |
| `LD2410_BAUD_INDEX_256000`  | 256000 *(factory default)* |
| `LD2410_BAUD_INDEX_460800`  | 460800 |

```cpp
// 1. Drop to 115200 (more tolerant on noisy CP2102 USB-UART clones)
radar.setBaudRate(LD2410_BAUD_INDEX_115200);
radar.requestRestart();
RADAR_SERIAL.begin(115200, SERIAL_8N1, RX, TX);

// 2. Round-trip 256000 -> 115200 -> 256000 (the bench test pattern)
radar.setBaudRate(LD2410_BAUD_INDEX_115200); radar.requestRestart();
RADAR_SERIAL.begin(115200, SERIAL_8N1, RX, TX);
radar.requestFirmwareVersion();          // verify
radar.setBaudRate(LD2410_BAUD_INDEX_256000); radar.requestRestart();
RADAR_SERIAL.begin(256000, SERIAL_8N1, RX, TX);

// 3. Bump to 460800 if your wiring is short and clean
radar.setBaudRate(LD2410_BAUD_INDEX_460800);
radar.requestRestart();
RADAR_SERIAL.begin(460800, SERIAL_8N1, RX, TX);
```

> The LD2410S does NOT support `setBaudRate` — its baud is fixed at
> 115200 and changing it requires reflashing the radar firmware.

## Bluetooth

The LD2410C carries a BLE radio used by HLK's Android tuning app.
You can enable / disable it, change the pairing password, or query the
MAC address from UART.

### `bool setBluetooth(bool on)`

Toggle Bluetooth on or off. Non-volatile, takes effect after
`requestRestart()`.

```cpp
// 1. Disable to save power and remove an attack surface
radar.setBluetooth(false);
radar.requestRestart();

// 2. Re-enable for a tuning session, then turn it back off
radar.setBluetooth(true);  radar.requestRestart();
// ... pair phone, tune ...
radar.setBluetooth(false); radar.requestRestart();

// 3. Toggle from a button
if (digitalRead(BTN) == LOW) {
  radar.setBluetooth(!btState); btState = !btState; radar.requestRestart();
}
```

Example: [`../examples/bluetoothControl/bluetoothControl.ino`](../examples/bluetoothControl/bluetoothControl.ino).

### `bool requestMACAddress()`

Read the radar's BLE MAC. Populates `radar.mac_address[6]` in
**wire (big-endian) order** — print byte 0 first.

```cpp
// 1. Print the MAC for inventory
if (radar.requestMACAddress()) {
  for (uint8_t i = 0; i < 6; i++) {
    if (radar.mac_address[i] < 0x10) Serial.print('0');
    Serial.print(radar.mac_address[i], HEX);
    if (i + 1 < 6) Serial.print(':');
  }
  Serial.println();
}

// 2. Compare against a whitelist
const uint8_t expected[] = {0xE8,0xFF,0x21,0x7C,0xDA,0x7E};
radar.requestMACAddress();
if (memcmp(radar.mac_address, expected, 6) != 0) handleUnknownDevice();

// 3. Use as a unique device-ID source for MQTT topic naming
radar.requestMACAddress();
char topic[40];
snprintf(topic, sizeof(topic), "radar/%02x%02x%02x/state",
         radar.mac_address[3], radar.mac_address[4], radar.mac_address[5]);
```

Example: [`../examples/bluetoothControl/bluetoothControl.ino`](../examples/bluetoothControl/bluetoothControl.ino).

### `bool obtainBluetoothPermissions(const uint8_t password[6])`

Send the BLE pairing password (default `"HiLink"`). **Note:** per the
HLK protocol PDF, the radar's ACK to this opcode is delivered **only
over BLE**, not on UART. So this method usually returns `false` even
when the password is accepted. Treat false as "no UART feedback,
expected" rather than as a hard failure.

```cpp
// 1. Try the factory default
const uint8_t pwd[6] = {'H','i','L','i','n','k'};
radar.obtainBluetoothPermissions(pwd);   // return value typically false

// 2. Try a custom password set previously via setBluetoothPassword
const uint8_t pwd2[6] = {'M','y','C','o','d','e'};
radar.obtainBluetoothPermissions(pwd2);

// 3. Probe several known passwords (unusual; mostly for recovery)
const uint8_t* pool[] = {pwd, pwd2 /*, ... */};
for (auto p : pool) radar.obtainBluetoothPermissions(p);
```

### `bool setBluetoothPassword(const uint8_t password[6])`

Change the BLE pairing password to a new 6-byte value. Persists
across reboots. ACK arrives on UART.

```cpp
// 1. Change to a custom password
const uint8_t newPwd[6] = {'M','y','C','o','d','e'};
radar.setBluetoothPassword(newPwd);

// 2. Reset to factory default
const uint8_t hilink[6] = {'H','i','L','i','n','k'};
radar.setBluetoothPassword(hilink);

// 3. Generate from device-unique source
radar.requestMACAddress();
uint8_t pwd[6];
memcpy(pwd, &radar.mac_address[0], 6);
radar.setBluetoothPassword(pwd);
```

> ⚠️ The bench tests deliberately do **not** call `setBluetoothPassword`
> because there's no getter to verify the value back, so a mistake
> would silently lock you out of the BLE app. Be intentional.

## Distance resolution

The LD2410C can quantise distance two ways:

| Constant | Per-gate | Max range (9 gates) | Use case |
|---|---|---|---|
| `LD2410_DISTANCE_RESOLUTION_075M` | 0.75 m | 6.75 m | Default. General-purpose room scale. |
| `LD2410_DISTANCE_RESOLUTION_02M`  | 0.20 m | 1.80 m | Near-field, finer granularity. |

Non-volatile. Takes effect after `requestRestart()`.

### `bool setDistanceResolution(uint16_t resolution_index)`

Set the active mode. Use the constants above.

### `bool requestDistanceResolution()`

Read the current mode, populating `radar.distance_resolution`.

```cpp
// 1. Switch to fine-grained near-field, restart, verify
radar.setDistanceResolution(LD2410_DISTANCE_RESOLUTION_02M);
radar.requestRestart();
radar.requestDistanceResolution();
Serial.print("now in mode: "); Serial.println(radar.distance_resolution);

// 2. Read at boot and adapt UI accordingly
radar.requestDistanceResolution();
const float meters_per_gate =
  (radar.distance_resolution == LD2410_DISTANCE_RESOLUTION_02M) ? 0.20f : 0.75f;

// 3. Round-trip flip + restore (test pattern)
radar.requestDistanceResolution();
const uint16_t orig = radar.distance_resolution;
const uint16_t flip = (orig == LD2410_DISTANCE_RESOLUTION_075M)
                       ? LD2410_DISTANCE_RESOLUTION_02M
                       : LD2410_DISTANCE_RESOLUTION_075M;
radar.setDistanceResolution(flip); radar.requestRestart();
// ... measure ...
radar.setDistanceResolution(orig); radar.requestRestart();
```

Example: [`../examples/distanceResolution/distanceResolution.ino`](../examples/distanceResolution/distanceResolution.ino).
