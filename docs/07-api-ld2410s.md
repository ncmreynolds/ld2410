# 07 — LD2410S extensions

Methods exposed **only** on the LD2410S variant. To make these
visible at compile time, define `LD2410_VARIANT_S` before
`#include <ld2410.h>` (or pass `-DLD2410_VARIANT_S` as a build flag).

For methods on **all** variants see [`04-api-core.md`](04-api-core.md).

> ⚠️ **Status: UNVERIFIED on hardware.** The S code path was developed
> against the V1.00 protocol PDF and reviewed for correctness, but the
> maintainer's bench currently has no LD2410S sample. The host test
> suite covers the byte-level parsing logic and the CI compile matrix
> builds for S on ESP32 / ESP8266 / RP2040. The per-gate frame layout
> assumption (64-byte block split as 16 motion + 16 stationary,
> mirroring base/C convention) is informed but not measured. If you
> have an LD2410S and find a discrepancy, please open an issue with a
> captured frame dump.

The S extensions are organised around four feature clusters:

- **Generic parameter set** — six per-radar settings in one TLV write
- **Per-gate trigger / hold thresholds** — finer control than base/C
- **Auto-threshold tuning** — the radar self-tunes against ambient noise
- **Serial number** read/write
- **Output-mode switch** — standard 70-byte frame vs minimal 5-byte frame

---

## Generic parameter set

A single command (opcode 0x70) writes six settings together. The
matching read (0x71) populates six public fields.

| Field | Meaning |
|---|---|
| `detect_farthest_gate` | Far edge of the detection window (gate index, 0..15) |
| `detect_nearest_gate` | Near edge of the detection window |
| `unmanned_delay_s` | Inactivity timeout in seconds |
| `status_report_freq` | Status frame rate code (HLK §2.2.7) |
| `distance_report_freq` | Distance frame rate code |
| `response_speed` | Response speed code (lower = smoother, higher = snappier) |

### `bool writeGenericParameters(uint8_t farthest, uint8_t nearest, uint16_t delay_s, uint8_t status_freq, uint8_t distance_freq, uint8_t speed)`

```cpp
// 1. Standard "balanced" preset
radar.writeGenericParameters(15, 0, 5, 1, 1, 5);

// 2. Tight near-field (close-range presence detection)
radar.writeGenericParameters(5, 0, 2, 1, 1, 7);

// 3. Long-range monitoring with slow response
radar.writeGenericParameters(15, 3, 30, 1, 1, 3);
```

### `bool requestGenericParameters()`

Returns `true` on ACK and populates the six public fields above.

```cpp
// 1. Snapshot current state
radar.requestGenericParameters();
Serial.printf("range g%u..g%u, idle %us\n",
              radar.detect_nearest_gate, radar.detect_farthest_gate,
              radar.unmanned_delay_s);

// 2. Save / modify / restore pattern
radar.requestGenericParameters();
const uint16_t saved_idle = radar.unmanned_delay_s;
radar.writeGenericParameters(15, 0, 60, 1, 1, 5);  // 60 s for test
// ... do test ...
radar.writeGenericParameters(15, 0, saved_idle, 1, 1, 5);

// 3. Verify a write took effect
radar.writeGenericParameters(10, 2, 5, 1, 1, 5);
radar.requestGenericParameters();
if (radar.detect_farthest_gate != 10) Serial.println("write rejected");
```

## Per-gate trigger / hold thresholds

The S exposes two per-gate threshold arrays of 16 values (one per
gate). "Trigger" is the energy level required to start reporting a
target on that gate; "hold" is the level required to keep reporting
once triggered.

| Method | Opcode | Effect |
|---|---|---|
| `writeTriggerThresholds(uint8_t[16])` | 0x72 | Set the trigger array in one call |
| `requestTriggerThresholds()` | 0x73 | Read into `radar.trigger_thresholds[16]` |
| `writeHoldThresholds(uint8_t[16])` | 0x76 | Set the hold array in one call |
| `requestHoldThresholds()` | 0x77 | Read into `radar.hold_thresholds[16]` |

```cpp
// 1. Apply a uniform trigger threshold across all 16 gates
uint8_t t[16];
for (uint8_t g = 0; g < 16; g++) t[g] = 50;
radar.writeTriggerThresholds(t);

// 2. Linear ramp: stricter on far gates
uint8_t t2[16];
for (uint8_t g = 0; g < 16; g++) t2[g] = 30 + g * 4;
radar.writeTriggerThresholds(t2);

// 3. Read, modify one gate, write back
radar.requestTriggerThresholds();
radar.trigger_thresholds[3] = 25;                 // make gate 3 looser
radar.writeTriggerThresholds(radar.trigger_thresholds);
```

The same patterns apply to `writeHoldThresholds` /
`requestHoldThresholds` with `radar.hold_thresholds[16]`.

## Auto-threshold tuning

The S can spend a configurable scanning window observing ambient
energy and derive sensible thresholds automatically.

### `bool autoUpdateThreshold(uint16_t trigger_factor, uint16_t retention_factor, uint16_t scanning_time_s)`

Start a tuning session. The radar scans for `scanning_time_s`
seconds, then sets thresholds derived from observed noise +
`trigger_factor` / `retention_factor` margins.

The HLK protocol does **not** document a UART ACK for this opcode;
the library tolerates firmwares that do (4-byte standard envelope)
and reports `true` if one arrives, but the meaningful progress
indicator is the **0x03 progress frame** parsed automatically.

```cpp
// 1. Default: 2× trigger margin, 1× retention margin, 120 s scan
radar.autoUpdateThreshold(2, 1, 120);

// 2. Aggressive: tight margins, short scan (less reliable)
radar.autoUpdateThreshold(1, 1, 30);

// 3. Conservative: wide margins, long scan (best for noisy environments)
radar.autoUpdateThreshold(3, 2, 300);
```

### `uint16_t autoThresholdProgress()` / `bool autoThresholdReceived()`

`autoThresholdProgress()` returns the latest "progress × 100" value
parsed from a 0x03 progress frame (e.g. 5000 = 50.00%). Use
`autoThresholdReceived()` to know if any progress frame has arrived
yet (sticky flag — flips true on first frame, never resets).

```cpp
// Polling progress while a session is running
radar.autoUpdateThreshold(2, 1, 120);
while (true) {
  radar.read();
  if (radar.autoThresholdReceived()) {
    Serial.print(radar.autoThresholdProgress() / 100.0f);
    Serial.println(" %");
    if (radar.autoThresholdProgress() >= 10000) break;   // 100.00 %
  }
  delay(500);
}
```

## Serial number

### `bool writeSerialNumber(const uint8_t sn[8])`

Set the radar's 8-byte serial number. Persisted across reboots.

```cpp
// 1. Set a custom SN
const uint8_t sn[8] = {'F','o','o','-','0','0','1','A'};
radar.writeSerialNumber(sn);

// 2. Set from a hash of MAC address (custom inventory scheme)
uint8_t sn2[8] = {0};                 // ... compute from MAC ...
radar.writeSerialNumber(sn2);

// 3. Reset to a known default
const uint8_t sn3[8] = {'D','E','F','A','U','L','T','0'};
radar.writeSerialNumber(sn3);
```

### `bool requestSerialNumber()`

Read into `radar.serial_number[8]`.

```cpp
// 1. Print as a string
radar.requestSerialNumber();
char buf[9] = {0};
memcpy(buf, radar.serial_number, 8);
Serial.println(buf);

// 2. Print as hex
radar.requestSerialNumber();
for (uint8_t i = 0; i < 8; i++) Serial.print(radar.serial_number[i], HEX);

// 3. Verify a SN write took effect
radar.writeSerialNumber(sn);
radar.requestSerialNumber();
if (memcmp(radar.serial_number, sn, 8) != 0) Serial.println("SN mismatch");
```

## Output mode

The S can emit two different data frames:

- **Standard** (default): 70-byte intra payload with the full target
  state + per-gate energy block. Same envelope as base/C 0x01 frames.
- **Minimal**: 5-byte fixed-size frame `6E STATE DIST_LO DIST_HI 62`
  for low-bandwidth pipes. The library parses both transparently.

### `bool setOutputMode(bool standard)`

Switch the active mode. `true` = standard, `false` = minimal.

```cpp
// 1. Default — request standard frames (full data)
radar.setOutputMode(true);

// 2. Switch to minimal for a low-bandwidth wireless bridge
radar.setOutputMode(false);

// 3. Toggle on demand
static bool useStandard = true;
useStandard = !useStandard;
radar.setOutputMode(useStandard);
```

In minimal mode the 5-byte frame carries only the target type and
the object distance. `parse_minimal_frame_()` therefore:

- **Updates** `target_type_` and `detection_distance_` on every frame.
- **Zeroes** the per-track fields (`moving_target_distance_`,
  `moving_target_energy_`, `stationary_target_distance_`,
  `stationary_target_energy_`) so the per-track getters
  (`movingTargetDistance()`, etc.) return 0 instead of stale standard-
  frame values.
- **Does NOT clear** the per-gate energy arrays
  (`engineering_motion_energy_[]` / `engineering_stationary_energy_[]`).
  If you toggle into minimal mode after running standard mode for a
  while, `movingEnergyAtGate(g)` / `stationaryEnergyAtGate(g)` /
  `snapshotEngineeringMotionEnergies()` will keep returning the last
  values seen in standard mode until you switch back. Treat the per-
  gate accessors as **stale** while in minimal mode and gate them on
  whether you're actually emitting standard frames.
